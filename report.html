01 Executive Summary

KH Brokers is a modern e-commerce brokerage marketplace built with Next.js 16 (App Router), React 19, TypeScript (strict), and Tailwind CSS v4. The platform implements a dual-marketplace model with separate buyer and seller flows, featuring config-driven content, Supabase authentication, Mailchimp email marketing, and a polished responsive UI.

The project is currently in MVP stage — the frontend is feature-complete with a well-organized component architecture, but the backend relies on mock data for deals and lacks several production essentials (rate limiting, input validation, error tracking, testing).

03 Architecture & Project Structure

The project follows a config-driven architecture where all page content lives in TypeScript config files (src/config/*.config.ts). Components read from these configs, making content updates possible without touching component logic.

src/ ├── app/ # Next.js App Router │ ├── layout.tsx # Root layout (fonts, animations, suspense) │ ├── metadata.ts # Default SEO metadata │ ├── sitemap.ts # Dynamic sitemap generation │ ├── robots.ts # robots.txt rules │ ├── globals.css # Tailwind v4 theme + global styles │ ├── animations.css # Scroll & entrance animations │ ├── (auth)/ # Auth route group (no header/footer) │ │ ├── signin/ │ │ ├── signup/ │ │ └── demo/ │ ├── (main)/ # Marketplace route group │ │ ├── buyers/ │ │ ├── sellers/ │ │ ├── invest/ & invest-2/ │ │ ├── invest-success/ │ │ └── value-my-store/ │ ├── deals/ # Protected marketplace listings │ │ └── [id]/ # Dynamic deal detail │ ├── about/ contact/ # Info pages │ ├── privacy/ terms/ # Legal (theme-aware) │ └── api/ # API routes │ ├── auth/ (signup, signin, signout, me) │ └── health/ ├── components/ │ ├── ui/ # Primitives (Button, Container, etc.) │ ├── layout/ # Header, Footer, Conditional wrappers │ ├── auth/ # SignInForm, SignUpForm │ ├── modals/ # WelcomeModal, DealSpeakModal │ └── sections/ # Page sections by domain │ ├── buyers/ (13) │ ├── sellers/ (13) │ ├── deals/ (8) │ ├── invest/ (4) │ ├── value-my-store/ (3) │ └── legal/ (2) ├── config/ # 14 config files (content source) ├── lib/ # Utilities │ ├── supabase.ts # Lazy client factories │ ├── mailchimp.ts # Mailchimp API wrapper │ ├── animations.ts # Animation tokens │ └── utils.ts # cn() helper ├── types/ # Shared TypeScript types └── proxy.ts # Route protection (Next.js 16 proxy)

Key Architectural Patterns

  • Config → Component: All content in src/config/, components are pure renderers
  • Route Groups: (auth) hides header/footer; (main) for marketplace pages
  • Dual Theme: Buyers = purple (#a36af6), Sellers = green (#00965F)
  • Server Components by default: Only "use client" where needed (forms, interactivity)
  • Lazy initialization: Supabase clients created on first use, not at import time
  • Path alias: @/* maps to ./src/*

Route Protection — src/proxy.ts

Uses Next.js 16's proxy.ts convention (replaces deprecated middleware.ts). Protects /deals and /deals/* routes. If no auth cookies are present, redirects to /signin?redirect=/deals.

Limitation: The proxy only checks cookie existence, not token validity. A user with an expired token cookie will pass the proxy but fail at the API level. Consider adding token validation in the proxy or implementing a client-side redirect on 401.

07 Supabase Integration

Client Architecture

Two Supabase clients using lazy initialization pattern to avoid build-time crashes when env vars aren't available:

getSupabase() → createClient(URL, ANON_KEY)

// Admin client (bypasses RLS) — used in API routes
getSupabaseAdmin() → createClient(URL, SERVICE_ROLE_KEY)

08 Mailchimp Integration

On user signup, addToMailchimp() is called non-blocking (fire-and-forget) to subscribe the user to the Mailchimp audience. Signup will succeed even if Mailchimp fails.

09 Security Audit

Overall Security Rating: Needs Significant Improvement Multiple critical and high-priority issues must be addressed before production deployment.

Critical Issues

#IssueSeverityDetails
1 No Rate Limiting Critical Auth endpoints (/api/auth/signup, /api/auth/signin) have no rate limiting. Vulnerable to brute-force attacks and credential stuffing.
2 No Input Validation Critical API routes accept raw JSON without schema validation. No email format checks, no password strength enforcement, no field sanitization. Risk of injection attacks.
3 Email Auto-Confirmed Critical Signup uses email_confirm: true, skipping email verification. Anyone can create accounts with fake emails.
4 No CAPTCHA / Bot Protection High No reCAPTCHA, hCaptcha, or Turnstile on signup/signin forms. Bots can mass-register accounts.
5 Token Existence vs Validity High Proxy checks if cookies exist but doesn't verify if tokens are valid or expired.
6 No Refresh Token Rotation High Refresh tokens are not rotated after use. If a refresh token is stolen, it remains valid for the full 7-day lifetime.
7 No Password Reset Flow High No forgot-password endpoint. Users who forget their password are locked out permanently.
8 Missing Security Headers Medium No Content-Security-Policy, X-Frame-Options, or Permissions-Policy headers configured.
9 No Consent for Mailchimp Medium Users are auto-subscribed to Mailchimp without explicit opt-in checkbox. Potential GDPR/CAN-SPAM violation.
10 Secrets in .env.local Medium Secret keys stored in local env file. Must use hosting provider's secret management for production (Vercel Environment Variables, AWS Secrets Manager, etc.).

What's Done Well

  • Good httpOnly cookies prevent XSS token theft
  • Good SameSite=lax protects against basic CSRF
  • Good Secure flag enabled in production
  • Good Service role key is server-side only (not exposed to client)
  • Good Profile creation rollback on failure
  • Good Mailchimp errors don't block signup

15 Recommendations & Roadmap

Phase 1 — Before Launch (Critical)

  1. Add input validation — Use Zod schemas on all API routes to validate request bodies
  2. Implement rate limiting — Use Upstash or similar for auth endpoints (e.g., 5 attempts / minute)
  3. Enable email verification — Remove email_confirm: true, send verification email via Supabase
  4. Add forgot password flow — Implement /api/auth/reset-password using Supabase's built-in flow
  5. Set up error tracking — Integrate Sentry for server and client error monitoring
  6. Add error boundaries — Create error.tsx files in root and route groups
  7. Add security headers — CSP, X-Frame-Options, X-Content-Type-Options via next.config.ts
  8. Add Mailchimp consent checkbox — Required for GDPR/CAN-SPAM compliance
  9. Fix Mailchimp merge field tags — Verify all tags match Mailchimp audience configuration
  10. Move secrets to hosting provider — Use Vercel env vars or similar for production

Phase 2 — Post-Launch (High Priority)

  1. Build deal management backend — Supabase tables + CRUD API routes for deals
  2. Add admin dashboard — Manage deals, users, and content
  3. Integrate analytics — Google Analytics 4 or Plausible for usage insights
  4. Add user profile page — Let users edit their info after signup
  5. Set up testing — Vitest for unit tests, Playwright for E2E
  6. Optimize performance — Dynamic imports for Lottie, lazy-load video, image optimization
  7. Add loading skeletons — loading.tsx files for better perceived performance
  8. Add CAPTCHA — Cloudflare Turnstile or reCAPTCHA on auth forms

Phase 3 — Growth (Medium Priority)

  1. CMS integration — Replace config files with Sanity or Contentful
  2. Payment processing — Stripe integration for premium features
  3. Messaging system — Real-time buyer/seller communication
  4. OAuth providers — Google, LinkedIn sign-in
  5. CI/CD pipeline — GitHub Actions for automated testing and deployment
  6. Blog / content marketing — SEO-driven content for organic traffic
  7. Structured data — JSON-LD for deal listings (Product schema)
